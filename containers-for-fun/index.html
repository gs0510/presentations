The paper then goes on to define what a "systems program" looks like! The system program is likely to be used to support other software and applications programs, but may also be a complete applications package itself. for example, most application programs read from and write to files on disk. The operating system, written in a systems programming language like C provides this functionality. It is likely to be continuously evolving in the number and types of features it supports. This definition is fairly agreeable‚Äîcomputer systems are large-scale, long-used, and time-varying. The last entry to consider from 1970s is the paper "Systems Programming" from Donovan and folks. 1972 was a productive year! üòé What is systems programming? You may visualize a computer as some sort of beast that obeys all commands. It has been said that computers are basically people made out of metal or, conversely, people are computers made out of flesh and blood. However, once we get close to computers, we see that they are basically mSoftware Engineer - achines that follow very specific and primitive instructions. In the early days of computers, people communicated with them by on and off switches denoting primitive instructions. Soon people wanted to give more complex instructions. 1990s:Rise of Scripting languages

In the 70s and 80s, it seems like most researchers saw systems programming usually as a contrast to assembly programming. There simply were no other good tools to build systems However, in the mid 90s, a major sea change occurred in programming languages with the rise of dynamically-typed scripting languages.

1990s:

Rise of Scripting languages

  • Bash, languages like Perl, Python, Ruby, Javascript etc. worked their way into the mainstream.
  • Systems programming languages designed to build more primitive computer elements.
  • Scripting languages are designed for gluing.
‚ÄúScripting: Higher Level Programming for the 21st Century‚Äù (Ousterhout 1998) Scripting languages are designed for different tasks than system programming languages, and this leads to fundamental differences in the languages. System programming languages were designed for building data structures and algorithms from scratch, starting from the most primitive computer elements such as words of memory. In contrast, scripting languages are designed for gluing: they assume the existence of a set of powerful components and are intended primarily for connecting components together. System programming languages are strongly typed to help manage complexity, while scripting languages are typeless to simplify connections between components and provide rapid application development. On a technical level, Ousterhout contrasted scripting vs. systems along the axes of type-safety and instructions-per-statement. On a design level, he characterized the new roles for each language class: systems programming is for creating components, and scripting is for gluing them together.

What is a systems programming language today?

languages in the category of C, C++, Rust, and D are distinguished in terms of their level of abstraction from the machine. These languages expose details of the underlying hardware like memory allocation/layout and fine-grained resource management. when you have an efficiency problem, how much freedom do you have to solve it? The wonderful part of low-level programming languages is that when you identify an inefficiency, it is within your power to eliminate the bottleneck by careful control over machine details. In the same way static types provide more confidence like ‚Äúthese two things I‚Äôm trying to add are definitely integers,‚Äù low-level languages provide more confidence that ‚Äúthis code will execute on the machine as I specified.‚Äù Low-level programmers should seek to understand what ideas in systems design could be adapted to deal with the reality of modern hardware.

Where does Rust shine?

Rust community has been exceedingly innovative in this respect, looking at how good software design/functional programming principles can be applied to low-level problems (e.g. futures, error handling, or of course memory safety). We should also consider the additional benefits for systems programming that are enabled by Rust‚Äôs type system: software fault isolation (SFI), program analysis ‚Äì especially static information flow control (IFC), and safe traversal of pointer-linked data structures.

namespaces

  • Namespaces determine what a process can see.
  • Multiple namespaces: pid, net, uts, ipc, user, mnt
Linux namespaces were inspired by the wider namespace functionality used heavily throughout Plan 9 from Bell Labs.[1] Just as chroot allows processes to see any arbitrary directory as the root of the system (independent of the rest of the processes), Linux namespaces allow other aspects of the operating system to be independently modified as well. This includes the process tree, networking interfaces, mount points, inter-process communication resources and more. in a pid namespace you become PID 1 and then your children are other processes. All the other programs are gone in a networking namespace you can run programs on any port you want without it conflicting with what‚Äôs already running in a mount namespace you can mount and unmount filesystems without it affecting the host filesystem. So you can have a totally different set of devices mounted.

cgroups

  • Allow processes to be organized into hierarchical groups.
    • The resources and usage of these groups can be limited.
  • Implemented through a pseudo filesystem called cgroupfs.
Okay, so we‚Äôve made a new magical world with new processes and sockets that is separate from our old world. Cool! What if I want to limit how much memory or CPU one of my programs is using?

cgroups

  • Subgroups inherit the limits set on the parent group.
  • A process can be a part of only one cgroup.
  • A subsystem controls each resource, e.g. the memory subsystem controls the memory use.
If the process exceeds the memory limit, it can be OOM killed or will keep emitting OOM notifications.

Conclusion üòä

Containers aren‚Äôt magic. Anyone with a Linux machine can play around with them and tools like Docker and rkt are just wrappers around things built into every modern kernel. No, you probably shouldn‚Äôt go and implement your own container runtime. But having a better understanding of these lower level technologies will help you work with these higher level tools (especially when debugging). In conclusion, Rust is so much more empowering than C, or C++ for systems programming because it follows the adage if it compiles, it works. In systems programming, sometimes you must perform operations that cannot be statically verified as safe. Rust gives you the tools to wrap these operations in safe abstractions, meaning that what was once relegated to code comments or convention can be statically enforced by the Rust compiler. If you were to do the same in C/C++, there would be so many times you would have to take care of memory allocation, and freeing this memory and other nitty-gritties that come with working with C/C++ which means it not accessible and is intimidating for a lot of people. Empowering people to write programs that they otherwise would have not written is superpower that Rust has! Lastly, I would like to give a reminder that a lot of systems things aren't really that hard if explained in a way that makes sense. I think computer networking is a really good example of this -- a lot of people get really intimidated by networking, but a lot of the core concepts like IP addresses and ports and packets are not really that hard, and once you understand them you can learn a lot.
